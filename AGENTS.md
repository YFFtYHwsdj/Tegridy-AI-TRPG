# AI 跑团系统

## 项目

AI 主持的桌面角色扮演游戏。核心规则以 PBTA 为骨架，融合《异景 / :Otherscape》机制。
玩家用自然语言描述行动，所有规则翻译工作由多 Agent 系统在幕后完成。
**玩家只说人话。规则藏在 Agent 背后。**

## 技术

- Python 项目，本地 `.venv` 管理依赖，本地venv版本为3.11，请优先使用本地环境。
- LLM API：DeepSeek（OpenAI-compatible 格式）
- `prompt, rules and conpects/` 目录下为 Agent prompt 参考模板，包含大量规则信息，非运行时直接加载
- 代码质量：
  - [ruff](https://docs.astral.sh/ruff/) — lint 和格式化，配置在 `pyproject.toml` 的 `[tool.ruff]` 段
    - `ruff check .` — 检查代码质量
    - `ruff check --fix .` — 自动修复可修复的问题
    - `ruff format .` — 格式化代码
  - [pyright](https://github.com/microsoft/pyright) — 静态类型检查，配置在 `pyrightconfig.json`
    - `pyright .` — 运行类型检查
    - 检查模式：`basic`（非 strict，因为 Agent 流水线层大量使用松散 dict 传递数据，strict 会产生过多误报）
    - 检查范围：`src/` + `main.py`（tests 中的 mock 类型噪音不计入）
    - 要求：**源码零 error**，warning 可接受

## 核心设计原则

1. **叙事与规则分离** — 玩家侧零规则暴露。意图解析、标签匹配、行动拆分、效果分配、后果兑现全部在 Agent 流水线内部完成，最终只输出沉浸式叙事文本。
2. **多 Agent 协作** — 系统由裁判、效果推演、后果、叙述者、节奏等多个专门 Agent 组成，各司其职，接力完成从规则裁判到叙事渲染的完整流水线。
3. **效果推演优先于效果计算** — 不由玩家花费力量购买效果。效果推演 Agent 作为因果模拟器，思考"以这个角色的能力，在这个环境中做这件事，实际上会造成什么"，然后翻译为效果类型和等级。

## 关键编码约定

1. **可判定逻辑用代码，需理解判断用 LLM** — 力量计算、掷骰解析、状态叠加溢出、极限达成检测用纯 Python（确定性、零 token 消耗）；标签适用性判断、意图解析、效果推演、叙事生成由 LLM Agent 处理。
2. **Agent 间传递"分析便签"而非"表单"** — Agent 输出是带推理过程的自然语言分析。下一个 Agent 读到前一个的完整思考过程，做出更连贯的接力判断。固定 JSON 字段只在力量值、掷骰结果等真正需要的地方使用。
3. **场景作为上下文单元** — 游戏按场景推进。场景结束时压缩 Agent 将对话历史压缩为结构化摘要，下一场景以前情提要启动，避免上下文无限膨胀。
4. **每次修改代码后必须运行 ruff + pyright** — 任何 `.py` 文件修改完成后，必须依次执行 `ruff check .`、`ruff format .` 和 `pyright .`，确保 lint 零告警、源码类型检查零 error。这是强制步骤，不可跳过。
5. **测试与代码变更同步** — 任何修改核心逻辑、新增功能或修复 bug 的代码变更，必须同步更新或补充对应的测试。不可只改实现不改测试，也不可跳过测试运行。测试是代码契约的一部分，不是可选项。
  - **单元测试**：运行 `python -m unittest discover -s tests` 确保单元测试全部通过。
  - **E2E 测试**：项目包含完整的自动化端到端测试（`e2e_test/` 目录）。可通过 `python -m e2e_test` 运行由 AI 扮演玩家的自动化跑团流程，以验证核心循环、多 Agent 协作和叙事连贯性。修改关键流程后应确保 E2E 测试也能正常运行。常用参数如下：
    - `--max-rounds N`：最大回合数，即玩家输入次数（默认 150）
    - `--max-scenes N`：最大场景切换次数（默认 15）
    - `--player-history N`：AI 玩家叙事历史窗口大小（默认 50）
    - `--no-debug`：关闭调试输出（仅显示纯叙事文本）
6. **注释是代码的一部分，不是可选项** — 代码的可维护性取决于后来者（包括三个月后的自己）能否快速理解设计意图。注释应解释"为什么这么做"，而非复述代码"做了什么"。具体要求如下：
  - **模块级 docstring** — 每个 `.py` 文件必须有模块级 docstring，描述该模块在系统中的角色和职责。
  - **公开函数/方法 docstring** — 所有公开 API（非 `_` 前缀）必须有 docstring，说明参数含义、返回值和副作用。使用 Google style 或 NumPy style，项目内保持一致。
  - **复杂逻辑必须注释** — 涉及规则机制（如 TTRPG 的 tick 溢出、力量计算、状态叠加）的任何非显而易见逻辑，必须用行内或块注释说明设计意图和边界条件。
  - **不要求注释自解释代码** — 类型注解完整、命名清晰的简单 getter/setter、数据类字段等不需要画蛇添足的注释。
  - **注释语言** — 注释和 docstring 使用**中文**，与项目整体语言保持一致。
